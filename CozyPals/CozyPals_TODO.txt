CozyPals - Deep ToDo / Design + Research Plan
=============================================

Project goal
------------
Build a Palworld UE4SS-based mod that makes base pals feel socially alive.
The mod should give each individual pal a persistent personality, contextual dialogue,
relationship progression, morale/trust interaction effects, and small side-quest/fetch-request loops.

Core vision
-----------
1. Each pal should feel like an individual, not just a species template.
2. Dialogue should depend on species + type + random hidden traits + current mood/work conditions + trust.
3. Interactions should matter both emotionally and mechanically.
4. Trust should expand beyond vanilla combat stats into base-life effects.
5. Data must persist per individual pal across reloads/world sessions.
6. The mod should be architected so single-player is easy first, while dedicated/co-op support remains possible later.

PHASE 2 - PAL IDENTITY / PERSISTENT DATA MODEL
=====================================================================

Goal:
Make one individual pal recover the same custom personality/quest history after restart.

Primary design
--------------
Use a hybrid model:
- Runtime tables in Lua for active objects.
- Persistent external save file for custom mod data.
- Keyed by a stable per-pal ID if available.

[ ] Define runtime table.

    TrackedPals[pal_uid] = {
      actor = <live actor ref>,
      actor_class = "...",
      species = "...",
      level = 0,
      type = "...",
      derived = {
        current_trust = 0,
        current_mood_state = "...",
        current_work_state = "...",
        current_negative_condition = nil,
      },
      save = <persistent data record>,
      dirty = false,
      last_seen = 0,
    }

[ ] Define persistent data schema.

    pals[pal_uid] = {
      version = 1,
      species = "Lamball",
      personality = {
        seed = 12345,
        work_attitude = "diligent",
        social_preference = "loves_talking",
        temperament = "gentle",
      },
      relationship = {
        interaction_xp = 0,
        talk_count = 0,
        pet_count = 0,
        cheer_count = 0,
        quest_count_completed = 0,
        last_pet_time = 0,
        last_talk_time = 0,
        last_cheer_time = 0,
        last_quest_offer_time = 0,
      },
      quest = {
        active = false,
        quest_type = nil,
        item_id = nil,
        item_count = 0,
        rarity = 0,
        offered_at = 0,
        expires_at = 0,
        reward_trust = 0,
        reward_morale = 0,
      },
      flags = {
        introduced = false,
        seen_depressed_once = false,
        favorite_request_revealed = false,
      },
      meta = {
        first_seen = 0,
        last_seen = 0,
        home_base_id = nil,
      }
    }

[ ] Decide save-file granularity.
    Recommended:
    - one file per world/save slot
    - server-owned file for dedicated later

[ ] Implement data migration policy.
    If fields are missing on load:
    - fill defaults
    - preserve unknown fields

[ ] Add dirty-flag persistence system.
    Save triggers:
    - new pal discovered
    - personality rolled
    - interaction performed
    - quest assigned/completed/expired
    - periodic autosave

[ ] Decide serialization format.
    Preferred:
    - JSON for readability
    Alternate:
    - Lua table return file if easier

[ ] Define fallback UID strategy if stable pal GUID is not exposed.
    Fallback key candidates:
    - owner_uid + container_id + slot + species + first_seen
    - world_id + base_id + species + local spawn sequence
    Risk:
    - pals moving containers/slots can break identity

=====================================================================
PHASE 5 - TRUST EXPANSION DESIGN
=====================================================================

Vanilla trust note
------------------
Vanilla trust already affects combat-oriented stats. Our mod should add cozy/base-life effects
without making trust progression absurdly overpowered.

[ ] Decide trust model policy.
    Recommended:
    - preserve vanilla trust effects
    - add modded secondary effects on top

[ ] Define modded trust effects.
    Candidate effects:
    - work_speed_bonus
    - slower_SAN_loss
    - faster_mood_recovery
    - reduced_negative_condition_chance
    - better response to cheer_up
    - stronger quest reward multipliers
    - more affectionate dialogue unlocks

[ ] Determine implementation path.
    Options:
    1. Hook/modify trust calculation directly if accessible.
    2. Add derived buffs from trust at runtime.
    3. Hybrid approach.

[ ] Build trust bonus tables.
    Example design:
    - low trust: small/no bonus
    - mid trust: mild work and morale stability bonus
    - high trust: stronger reliability / recovery bonus
    - max trust: special behavior unlocks, not just more raw stats

[ ] Add balance limits.
    Suggested controls:
    - diminishing returns above certain trust thresholds
    - daily caps on social gain
    - quest cooldowns
    - interaction cooldowns

[ ] Decide whether trust can exceed vanilla cap only if 99-cap mod layer is active.
    Need to investigate:
    - can we safely extend trust cap ourselves
    - what UI or data assumptions break if trust > vanilla maximum

    - cycle through interaction modes

PHASE 7 - FETCH QUEST / REQUEST SYSTEM
=====================================================================

Vision
------
Instead of generic gifts, pals occasionally ask for specific attainable items.
This becomes the highest-trust-yield social loop.

[ ] Define quest types for v1.
    Start simple:
    - fetch item request only

[ ] Define item-pool tiers by pal level.
    Rules desired:
    - cap maximum request rarity by pal level
    - allow any lower-tier item under that cap
    - keep requests believable and attainable

[ ] Build rarity tiers.
    Example:
    - common
    - uncommon
    - rare
    - elite / very rare

[ ] Build item families.
    Candidate categories:
    - berries / food
    - cooked meals
    - natural materials
    - ingots / crafted goods
    - medicine
    - species-flavored rare drops (feathers, fur, etc.)

[ ] Map rarity to reward.
    Reward channels:
    - trust gain
    - morale gain
    - special dialogue
    - possibly temporary buff

[ ] Determine request generation rules.
    Suggested filters:
    - pal level
    - species/theme
    - type/element
    - current mood/need state
    - quest cooldown
    - trust threshold minimum

[ ] Determine request limits.
    Recommended:
    - 1 active request per pal
    - limit active requests globally per base
    - cooldown after completion or expiry

[ ] Determine request resolution UX.
    Need to answer:
    - does the player manually hand in with interaction?
    - does the game auto-detect item in inventory while talking?
    - do we require item stack count in inventory then consume?

[ ] Determine failure/expiry behavior.
    Suggested:
    - mild disappointment dialogue only
    - no harsh punishment
    - small cooldown before next request

[ ] Decide how favorite requests work.
    Future idea:
    - some species/traits love certain categories and reward more trust

FIRST MILESTONE DEFINITION
=====================================================================

Milestone 1: "One pal keeps its identity"
------------------------------------------
Success criteria:
[ ] Detect one specific base pal at runtime.
[ ] Read a stable unique ID (or best fallback key).
[ ] Roll and store a personality for that pal.
[ ] Save to disk.
[ ] Restart game/world.
[ ] Reload the same pal and recover the same personality.

Milestone 2: "One pal can talk"
--------------------------------
[ ] Trigger one custom interaction.
[ ] Show one contextual dialogue line.
[ ] Apply tiny trust/morale effect.
[ ] Persist interaction counts.

Milestone 3: "One pal can ask for something"
---------------------------------------------
[ ] Generate one fetch request based on pal level.
[ ] Detect inventory item turn-in.
[ ] Consume item.
[ ] Reward trust/morale.
[ ] Save completion history.

=====================================================================
FINAL NOTES
=====================================================================

Strong recommendation:
Do not write huge amounts of dialogue content until the following are proven:
1. pal identity is stable
2. trust/mood/work data can be read reliably
3. interaction hook points are practical
4. save/load works safely

Once those four are real, content production can ramp up very fast.
